So what makes a map?
Maps are made of styled geographic data.
There are many geographic data formats.

Let's go back in time.



In 1996, Mapquest launched its web service.
Directions for driving! Online! Wow!
The problem: it was slow to load :(.
Mapquest required a full page refresh to scroll or zoom and was always aligned to map image boundaries.
Then, in 2005, came Google Maps. This revolutionized everything.
But what was the revolution?
Was it the interface?
The red marker?
The weather widget?
No!
It was the tile.
In particular, the raster tile.
Raster tiles are 256px x 256px PNG files that are stored on the server and served to the browser on request.
Each tile is one piece of the map in a grid of pieces.
You may think of this as a basemap.
(This kind of map is colloquially called a slippy map.)
Only the tiles in the current map view are requested and loaded. This makes modern web maps super hella fast.
Raster tiles are available at preset zoom levels with a different set of tiles at each zoom.
Zoom level 0: one tile for the world.
Number of tiles = 2^z x 2^z.
Zoom level 1: 4 tiles for the world.



Zoom level 2, 3, 4, 5
These tiles are made with a super hella awesome cool open source tool called Mapnik.
Mapnik is what makes all of this possible.
What does Mapnik do?
Let's look at how raster tiles are made.
1. Use vector data (shapefile, GeoJSON, etc)
2. Style the data with a styling language called CartoCSS
3. Export data and stylesheet to a renderer (Mapnik) that creates tiled images.
The images are created at each zoom level and stored in a specific file structure as PNGs.
That makes sense, right?
Vector data + style = image with style applied to data.
(If you are familiar with TileMill, this is what that application did.)
Then, using a web mapping library like Leaflet, you create a map container that knows when to load which tiles from your tile server.
When your map needs the image tile for a particular area at a particular zoom, your map container (via the browser) requests it from the server.

This was innovative in 2005...
... but it's also limiting, especially with the introduction of retina displays.
With retina displays came the need for larger, denser map tiles...
... all while keeping support for non-retina screens.
That meant storing twice as many map tiles, which was a huge bummer!
And thus was born the vector tile.
Before I continue, are there any questions?
More recently, we've graduated to vector tiles.
Vector tiles are similar to raster tiles...
... but instead of creating tiled images, we create tiled vector data that can be rendered in the browser.
All of this still happens with Mapnik. All hail Mapnik.
Roads, parks, water bodies, bike paths, points of interest, state boundaries...
... all of this is vector data with attributes attached to it.
Vector tiles store all of this information, attributes included, as protocol buffers, or PBF files.
Protocol buffers are super lightweight and compact.
To be quite honest, this doesn't change things on the client-side all that much.
Your maps are still made up of raster tiles.
BUT
What has changed is how they're being stored and generated.
What's different?
1. Use vector data
2. Convert the vector data from its original format to vector tile format (PBF, or protocol buffers).
3. Style the data with CartoCSS
4. When a style is requested in the browser, Mapnik takes the style, grabs the source it references, and creates image tiles ON THE FLY to serve to the browser.
Why is this beneficial?
A) Vector tiles are stored with all data attributes and can be styled with multiple styles.
B) This is a way more compact system for data storage (not having to store retina and non-retina, etc).
C) You can give Mapnik options on request (e.g. retina, multiple styles) and the tiles are created and served quickly .
D) Hosting vector data independently of the style means it can be rendered in many different ways...
... which has enabled one of the neatest innovations in map creation:
GL!
Some of you may have seen or heard of this crazy GL thing.
GL is a different way of rendering data on your screen.
With both older raster tile creation and the newer vector tile creation process, your map is requesting images -- slower.
With GL, instead of passing the images to the browser...
... you will actually pass the vector data itself to the browser...
... where the WebGL processor will render the graphics right in your browser -- faster.
This technology is in its infancy.
We probably won't talk about it too much throughout the course
but I wanted you to know that it exists before moving forward.
